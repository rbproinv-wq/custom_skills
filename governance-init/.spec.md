# .spec.md — governance-init: Refactoring to Interactive Multi-Stack Scaffolder

## 1. Objective

Transform `governance-init` from a minimal 44-line system prompt into a **complete project initialization skill** with interactive setup, multi-language scaffolding, behavioral governance, and post-init validation. Leverage patterns from 6 community skills identified as relevant.

## 2. Key Changes

| Current | Future |
|---------|--------|
| 44-line system prompt with 7 rules | Full SKILL.md with workflow, edge cases, checklist, output format |
| Static `init.sh` — copies templates blindly | Interactive wizard — detects project type, asks questions, scaffolds language-specific + governance files |
| `validate_env.sh` — checks 3 API keys | Robust prerequisite validation — OS, language runtimes, package managers, API keys, port availability |
| Generic `.crush.md` template | Context-aware `.crush.md` — tech-stack-specific rules, BFRI scoring, behavioral governance |
| `crush.json` with missing cache fields | Schema-compliant `crush.json` with `cost_per_1m_in_cached` / `cost_per_1m_out_cached` |
| Hardcoded model references (GLM-4.6) | Updated model references (current-gen models) |
| No validation after init | Post-init checklist verifying all artifacts were generated correctly |

## 3. Features Added (from Community Skills)

| Feature | Source Skill | Implementation |
|---------|-------------|----------------|

| Prerequisite checking | `web-artifacts-builder`, `environment-setup-guide` | `validate_env.sh` now checks language runtimes (node/python/go/rust), ports, git config |

## 5. Interactive Wizard Flow (`init.sh --wizard`)

```
Welcome to Crush Governance Init!
-----------------------------------
1. Project type: [API / Fullstack / Library / CLI / Mobile]
2. Language: [Python / TypeScript / Go / Rust]
3. Framework (if applicable): [FastAPI / Next.js / Express / Gin]
4. Database: [PostgreSQL / SQLite / MongoDB / None]
5. API keys setup: [DeepSeek / Gemini / OpenRouter / Skip]
6. LLM Router: [local / none]
7. Devcontainer: [yes / no]

→ Generates governance files + language-specific templates
→ Runs validate.sh to confirm everything is in place
```

## 6. Edge Cases

| Scenario | Handling |
|----------|----------|
| User runs init in empty directory | Detects no existing files → generic setup |
| User runs init in existing project | Detects `package.json`/`pyproject.toml`/`go.mod` → infers stack, asks to confirm |
| No git user config | Checks `git config user.name`/`user.email`, warns if missing |
| Port 8443 already in use | Detects and warns before configuring LLM Router |
| API keys missing | Validates but doesn't fail — marks as warning, router will fail gracefully |
| Template directory missing | Falls back to generic template |
| User cancels wizard mid-way | No files written (uses temp dir, copies only on confirm) |

## 8. Acceptance Criteria

1. `init.sh --wizard` interactively scaffolds a complete project with governance + language-specific files
2. `init.sh --detect` correctly identifies existing project type from `package.json`/`pyproject.toml`/`go.mod`
3. `validate_env.sh` checks runtimes (node/python/go), ports, git config, AND API keys
4. `validate.sh` (post-init) verifies all expected files exist and are non-empty
5. `crush.json` template has zero LSP warnings about missing fields
6. `.crush.md` includes behavioral governance (3 Iron Rules)
7. SKILL.md has all standard sections (When to Use, Workflow, Edge Cases, Checklist, Output)
8. Running init twice in same directory is safe (no duplicate files, no errors)

## 9. Non-Goals

- Not creating actual runtimes or installing dependencies (just scaffolding)
- Not adding CI/CD pipeline generation
- Not migrating existing projects into Crush format (only new setup)
- Not modifying the LLM Router itself (just configuration)
