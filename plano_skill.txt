================================================================================
1. OBJETIVO GERAL
================================================================================

Desenvolver aplicações completas (backend, frontend, scripts, automações) usando
um agente de IA no terminal (Crush) gastando o MÍNIMO POSSÍVEL em APIs, sem
perder qualidade de código. Almeja-se um padrão similar ao Claude Code, mas com
custo 50 a 100 vezes menor.

Abordagem: roteamento inteligente de tarefas para modelos baratos, uso agressivo
de prompt caching, e configuração multi-provedor com modelos de alto
custo-benefício (Gemini Flash-Lite, DeepSeek-V3, GLM-4.6, GPT-4.1-nano).

================================================================================
2. FERRAMENTA PRINCIPAL: CRUSH
================================================================================

Nome: Crush
Desenvolvedor: Charm (charmbracelet)
Tipo: Agente de IA para terminal (similar a Claude Code, Aider)
Licença: Open source
Instalação: brew install charmbracelet/tap/crush (macOS) ou npm install -g @charmland/crush
Funcionalidades-chave:
- Suporte a múltiplos provedores (OpenAI-compat, Gemini, Anthropic, etc.)
- Arquivos de contexto automáticos (.crush.md, AGENTS.md, CLAUDE.md)
- Modo agente com ferramentas (view, edit, grep, bash)
- Sistema de custos integrado (cost tracking)
- Cache de contexto automático (depende do provedor)
- Permissões granulares
- TUI (terminal user interface) baseada em BubbleTea

Por que o Crush?:
- Flexibilidade de modelos (não fica preso a um único provedor)
- Baixo consumo de recursos locais (não exige GPU, roda com 4-8GB RAM)
- Integração nativa com MCP (Model Context Protocol) para extensões
- Transparência de custos por token

================================================================================
3. ESTRATÉGIA DE ECONOMIA (3 PILARES)
================================================================================

PILAR 1 - ROTEAMENTO INTELIGENTE DE MODELOS
-----------------------------------------------------------------------------
Conceito: Cada tarefa de desenvolvimento tem um nível de complexidade. Usar um
modelo caro (ex: Claude Opus) para gerar boilerplate é desperdício. Usar um
modelo fraco para debugar um algoritmo complexo leva a retrabalho.

Implementação:
- Tarefas SIMPLES (boilerplate, documentação, testes unitários, formatação) →
  modelo MAIS BARATO (Gemini Flash-Lite, GPT-4.1-nano)
- Tarefas MÉDIAS (lógica de negócio, CRUD, integrações simples) →
  modelo CUSTO-BENEFÍCIO (DeepSeek-V3)
- Tarefas COMPLEXAS (otimização de algoritmo, arquitetura, segurança avançada) →
  modelo PREMIUM (GLM-4.6, raramente Claude Sonnet)

Resultado esperado: redução de custo de 80-95% em relação a usar um único modelo
premium para tudo. Pesquisas mostram que roteamento pode cortar custos em 95,1%
sem perda de qualidade (arXiv:2508.02216).

PILAR 2 - MAXIMIZAÇÃO DO PROMPT CACHING
-----------------------------------------------------------------------------
Conceito: Muitos provedores (DeepSeek, Anthropic, Google) oferecem cache de
contexto: a primeira vez que um bloco de texto (ex: regras do projeto) é enviado,
ele é armazenado; nas próximas vezes, o custo por token é reduzido em até 90%.

Implementação:
- Estruturar o prompt com um "BLOCO ESTÁVEL" (sempre igual) no INÍCIO da mensagem.
- Esse bloco contém: system prompt, regras de codificação, padrões de projeto,
  caminhos de diretórios, exemplos de estilo.
- O bloco DINÂMICO (tarefa atual) vai no final.
- Manter a mesma sessão do Crush para múltiplas tarefas relacionadas, para que
  o cache persista.

Exemplo de estrutura:
```
# === BLOCO ESTÁVEL (CACHED) ===
Você é um engenheiro sênior em Python.
Regras:
- Use type hints
- Docstrings no formato Google
- Máximo 30 linhas por função

# === BLOCO DINÂMICO ===
Implemente uma função que valide emails.
```

Benefício: economia de 45-80% nos custos de API e latência reduzida em 13-31%
(arXiv:2601.05871).

PILAR 3 - SELEÇÃO DE MODELOS COM MELHOR CUSTO-BENEFÍCIO
-----------------------------------------------------------------------------
Conceito: O mercado de LLMs está cada vez mais competitivo. Modelos como Gemini
Flash-Lite e DeepSeek-V3 oferecem qualidade surpreendente por fração do preço
de Claude ou GPT-4.

Critérios de escolha:
- Preço por 1M tokens (input e output)
- Existência e desconto de prompt caching
- Context window (mínimo 32K para código)
- Performance em benchmarks de código (HumanEval, SWE-bench)

5. ARQUITETURA DE CONFIGURAÇÃO (ARQUIVOS)
================================================================================

Para cada projeto, devem existir os seguintes arquivos na raiz:

5.1. crush.json (configuração de provedores e modelos)
-----------------------------------------------------------------------------
Exemplo completo:

```json
{
  "$schema": "https://charm.land/crush.json",
  "providers": {
    "deepseek": {
      "type": "openai-compat",
      "name": "DeepSeek",
      "base_url": "https://api.deepseek.com/v1",
      "api_key": "$DEEPSEEK_API_KEY",
      "models": [
        {
          "id": "deepseek-chat",
          "name": "DeepSeek-V3",
          "cost_per_1m_in": 0.27,
          "cost_per_1m_out": 1.10,
          "cost_per_1m_in_cached": 0.07,
          "context_window": 64000
        }
      ]
    },
    "google-gemini": {
      "type": "gemini",
      "name": "Google Gemini",
      "api_key": "$GEMINI_API_KEY",
      "models": [
        {
          "id": "gemini-2.0-flash-lite",
          "name": "Gemini 2.0 Flash-Lite",
          "cost_per_1m_in": 0.075,
          "cost_per_1m_out": 0.30
        }
      ]
    },
    "openrouter": {
      "type": "openai-compat",
      "name": "OpenRouter",
      "base_url": "https://openrouter.ai/api/v1",
      "api_key": "$OPENROUTER_API_KEY"
    }
  },
  "options": {
    "theme": "dark"
  }
}
```

Variáveis de ambiente (definir no ~/.bashrc ou .env):
- DEEPSEEK_API_KEY
- GEMINI_API_KEY
- OPENROUTER_API_KEY

5.2. .crush.md (regras e contexto estáveis para cache)
-----------------------------------------------------------------------------
Arquivo markdown com duas seções: BLOCO ESTÁVEL e BLOCO DINÂMICO.
Exemplo mínimo:

```markdown
# === BLOCO ESTÁVEL (CACHED) ===
## ROLE
Você é um engenheiro de software sênior especializado em [TECNOLOGIA].

## REGRAS OBRIGATÓRIAS
- Use type hints em todas as funções
- Documente com docstrings no formato Google
- Máximo de 30 linhas por função
- Testes unitários para toda lógica nova

## PADRÕES
- Python: black + isort + mypy
- JavaScript: ESLint + Prettier

# === BLOCO DINÂMICO ===
## TAREFA ATUAL
[Descreva aqui a tarefa específica]
```

5.3. AGENTS.md (alternativo, também lido pelo Crush)
-----------------------------------------------------------------------------
Pode conter instruções de mais alto nível para a equipe e IA.

5.4. Estrutura adicional opcional:
.crush/
├── prompts/
│   ├── review.md
│   ├── refactor.md
│   └── test.md
└── sessions/ (logs)

================================================================================
6. WORKFLOW DE DESENVOLVIMENTO (5 FASES)
================================================================================

FASE 0: SETUP INICIAL (primeira vez no projeto)
-----------------------------------------------------------------------------
- Executar script de automação `crush-init` (ver seção 8)
- Ou criar manualmente os arquivos crush.json, .crush.md, AGENTS.md
- Garantir que as variáveis de ambiente das API keys estão definidas

FASE 1: PLANEJAMENTO E DOCUMENTAÇÃO (custo muito baixo)
-----------------------------------------------------------------------------
Modelo: Gemini 2.0 Flash-Lite (ou GPT-4.1-nano)
Comando no Crush:
  > /model google-gemini/gemini-2.0-flash-lite
  > Crie um plano detalhado para a feature X, incluindo arquitetura e passos.

Resultado: docs/plano.md, docs/requisitos.md

FASE 2: DESENVOLVIMENTO DO CORE (custo baixo a médio)
-----------------------------------------------------------------------------
Usar roteamento manual:
- Tarefas simples (criar arquivos base, schemas, rotas):
    > /model google-gemini/gemini-2.0-flash-lite
- Tarefas de lógica complexa (algoritmos, validações, segurança):
    > /model deepseek/deepseek-chat

Dica: Manter a mesma sessão do Crush para aproveitar o cache.

FASE 3: REFINAMENTO E TESTES (custo muito baixo)
-----------------------------------------------------------------------------
Modelo: Gemini 2.0 Flash-Lite
Comando:
  > /model google-gemini/gemini-2.0-flash-lite
  > Gere testes unitários para todas as funções do arquivo auth.py.
  > Revise o código e sugira melhorias de legibilidade.

FASE 4: REFATORAÇÕES COMPLEXAS (custo médio, uso esporádico)
-----------------------------------------------------------------------------
Modelo: GLM-4.6 (via OpenRouter) ou DeepSeek-V3 se não houver GLM.
Comando:
  > /model openrouter/glm-4.6
  > Otimize a função de busca para O(log n) e adicione memoization.

FASE 5: REVISÃO FINAL (custo baixo)
-----------------------------------------------------------------------------
Usar dois modelos para revisão cruzada:
  1. Gemini Flash-Lite para verificar estilo e boas práticas.
  2. DeepSeek-V3 para análise de segurança e performance.
Combinar os resultados.

================================================================================
7. IMPLEMENTAÇÃO GRANULAR (PASSO A PASSO PARA UM PROJETO REAL)
================================================================================

Suponha um projeto: API FastAPI de gerenciamento de tarefas.

1. Abrir terminal, criar pasta:
   mkdir task-api && cd task-api

2. Executar automação (ou criar arquivos manualmente):
   crush-init

3. Editar .crush.md: trocar [TECNOLOGIA] por "Python/FastAPI/SQLAlchemy"

4. Iniciar o Crush:
   crush

5. Planejamento:
   > /model google-gemini/gemini-2.0-flash-lite
   > Crie um plano para API de tarefas: endpoints CRUD, modelo Task (id, titulo, descricao, concluida), banco SQLite.

6. Desenvolvimento do core (modelo e schemas):
   > /model google-gemini/gemini-2.0-flash-lite
   > Gere o modelo Task usando SQLAlchemy e o schema Pydantic.

7. Implementar lógica de negócio (validação de dados, regras):
   > /model deepseek/deepseek-chat
   > Implemente a lógica para não permitir tarefas com título duplicado. Adicione tratamento de erros.

8. Criar endpoints (tarefa simples):
   > /model google-gemini/gemini-2.0-flash-lite
   > Crie os endpoints CRUD completos usando FastAPI.

9. Gerar testes:
   > /model google-gemini/gemini-2.0-flash-lite
   > Gere testes com pytest para todos os endpoints.

10. Revisão de segurança:
    > /model deepseek/deepseek-chat
    > Revise o código quanto a injeção de SQL, validação de entrada, e autenticação.

11. Documentação final:
    > /model google-gemini/gemini-2.0-flash-lite
    > Gere um README.md com instruções de instalação, uso e exemplos de API.

12. Sair do Crush: Ctrl+C ou `exit`

CUSTO ESTIMADO TOTAL (para este projeto, com ~50K tokens processados):
- Planejamento + boilerplate: 10K tokens → Gemini: $0.00075
- Lógica complexa: 20K tokens → DeepSeek: $0.0054 (input) + $0.022 (output) ≈ $0.0274
- Testes + documentação: 20K tokens → Gemini: $0.0015
- Revisões: 10K tokens → Gemini + DeepSeek: $0.001
TOTAL: ~$0.03 USD (três centavos de dólar)

8. AUTOMAÇÃO DO SETUP: SKILL `crush-init`
================================================================================

Para evitar repetição manual, criamos um script bash chamado `crush-init` que
automatiza a criação da estrutura de arquivos.

8.1. Localização do script:
Salvar em /usr/local/bin/crush-init (ou qualquer diretório no PATH)
Dar permissão de execução: chmod +x /usr/local/bin/crush-init

8.2. Conteúdo do script crush-init:

```bash
#!/bin/bash
GREEN='\033[0;32m'
BLUE='\033[0;34m'
YELLOW='\033[1;33m'
NC='\033[0m'

echo -e "${BLUE}🚀 Inicializando projeto Crush...${NC}"

# Criar estrutura de pastas
mkdir -p .crush/prompts .crush/sessions

# Criar crush.json se não existir
if [ ! -f "crush.json" ]; then
    cat > crush.json << 'EOF'
{
  "$schema": "https://charm.land/crush.json",
  "providers": {
    "deepseek": {
      "type": "openai-compat",
      "name": "DeepSeek",
      "base_url": "https://api.deepseek.com/v1",
      "api_key": "$DEEPSEEK_API_KEY",
      "models": [
        {
          "id": "deepseek-chat",
          "name": "DeepSeek-V3",
          "cost_per_1m_in": 0.27,
          "cost_per_1m_out": 1.10,
          "cost_per_1m_in_cached": 0.07,
          "context_window": 64000
        }
      ]
    },
    "google-gemini": {
      "type": "gemini",
      "name": "Google Gemini",
      "api_key": "$GEMINI_API_KEY",
      "models": [
        {
          "id": "gemini-2.0-flash-lite",
          "name": "Gemini 2.0 Flash-Lite",
          "cost_per_1m_in": 0.075,
          "cost_per_1m_out": 0.30
        }
      ]
    }
  }
}
EOF
    echo -e "${GREEN}✅ Criado crush.json${NC}"
fi

# Criar .crush.md se não existir
if [ ! -f ".crush.md" ]; then
    cat > .crush.md << 'EOF'
# === BLOCO ESTÁVEL (CACHED) ===
## ROLE
Você é um engenheiro de software sênior.

## REGRAS OBRIGATÓRIAS
- Use type hints
- Documente com docstrings
- Máximo de 30 linhas por função
- Testes unitários para toda lógica nova

# === BLOCO DINÂMICO ===
## TAREFA ATUAL
[Descreva sua tarefa aqui]
EOF
    echo -e "${GREEN}✅ Criado .crush.md${NC}"
fi

# Criar AGENTS.md se não existir
if [ ! -f "AGENTS.md" ]; then
    cat > AGENTS.md << 'EOF'
# AGENTS.md - Instruções para IA

Este projeto utiliza roteamento inteligente de modelos:
- Tarefas simples: Gemini Flash-Lite
- Tarefas complexas: DeepSeek-V3

Consulte .crush.md para regras detalhadas.
EOF
    echo -e "${GREEN}✅ Criado AGENTS.md${NC}"
fi

# Criar prompts auxiliares
cat > .crush/prompts/review.md << 'EOF'
Revise o seguinte código quanto a: segurança, performance, boas práticas, bugs.
EOF

cat > .crush/prompts/refactor.md << 'EOF'
Refatore o código para melhor legibilidade e menor complexidade.
EOF

cat > .crush/prompts/test.md << 'EOF'
Gere testes unitários incluindo casos normais, borda e erro.
EOF

echo -e "${GREEN}✅ Estrutura completa criada!${NC}"
echo -e "${YELLOW}Próximos passos:${NC}"
echo "  1. Edite .crush.md com as regras específicas do projeto"
echo "  2. Execute 'crush' para iniciar o agente"
```

8.3. Como usar:
```bash
cd meu-novo-projeto
crush-init
crush
```

10. PRÓXIMOS PASSOS (A SEREM EXECUTADOS PELA TERCEIRA LLM)
================================================================================

Com base neste documento, a terceira LLM deve:

1. Validar e refinar a estratégia de roteamento, sugerindo melhorias ou novos
   modelos que tenham surgido no mercado.

2. Pesquisar e propor um padrão formal para criação de "skills" no Crush,
   inspirado no padrão Agent Skills (arquivos SKILL.md) ou MCP servers. O objetivo
   é transformar o `crush-init` em uma skill nativa, que possa ser invocada com
   um comando como `/skill init-project`.

3. Identificar as melhores práticas de engenharia de prompt para maximizar o
   cache em tarefas de desenvolvimento, incluindo técnicas de "prefix tuning"
   e "cache warming".

4. Elaborar um template de `SKILL.md` para o Crush, seguindo a documentação
   oficial (se existir) ou propondo uma estrutura padronizada.

5. Fornecer um guia de integração com MCPs (Model Context Protocol) para
   estender as capacidades do Crush, por exemplo, conectando a um banco de dados
   vetorial para cache semântico local.

6. (Opcional) Esboçar um sistema de monitoramento de custos que rode localmente,
   usando os logs do Crush e gerando alertas quando o gasto ultrapassar um
   limite configurável.

7. Responder com um plano de ação detalhado, incluindo código de exemplo,
   comandos e explicações, para que possamos implementar a skill de automação
   de setup de forma profissional e reutilizável.
